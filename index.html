2. Use GFMD in the PR description

  • GitHub Flavored Markdown (and Markdown in general) is useful in many places when using GitHub (e.g., issue tracker, PR reviews, writing documentation). The aim of this task is to ensure that you are sufficiently familiar with the GFMD syntax.

  • Requirements: Update the description of the iP PR you created earlier (do not add a new comment) so that it contains the following GFMD elements:

    1. a heading

    2. a bullet list

    3. a numbered list

    4. a fenced code block (with syntax highlighting)

    5. a task list

    6. an emoji

    7. a blockquote

    8. a hyperlink

    9. inline code

    10. some text formatting: bold, italic, strikethrough etc.

3. Review some peer PRs (please start on Friday in Week 5, instruction will be given during lecture)

Important

Please wait till Friday in Week 5 to start this task, to give others a few extra days to create the PR if they haven't done so yet.

You could refer to the textbook pages 204 - 207 (topic: Reviewing PRs)

 

Step 1: Note these additional guidelines:

  • Read the Best practices for reviewing PRs @SE-EDU/guides. You are expected to follow all of them.

  • Make sure you add 'review comments' (not regular comments) as only those are counted for participation. See step 4 in the panel above to find how to add 'review comments'.

  • If the PR has received some review comments already, feel free to confirm/complement/question those comments in your review. Also, look for things the previous reviewers may have missed.

  • At the end of the review, choose Comment (i.e., not Approve or Request changes)

 

Step 2: Do the first PR review as follows.

  • Give comment on coding standard related issues only.

  • Review comments don't always have to be about problems in the code. Other things you can do:

    • compliment the author on not making a common mistake

    • ask questions

    • suggest alternatives

Please refer to announcement that will be posted later on.

Note

Question: How many comments should I add?

Answer: Depends on the code being reviewed but we expect most PRs would warrant at least 4-5 comments. If the PR is huge, you can stop when you think you've put in a fair amount of time on the job (~15 minutes) and added enough comments for the PR author to receive some value.

 

Step 3: Do the second PR review as follows.

  • Comment on other code quality guidelines (see the section Guideline: Name well in textbook pages 81 to 83) you have learned so far. It's optional to comment on coding standard violations in this PR review.

Note

Question: What if the given sample Checkstyle config file doesn't match exactly with the Java coding standard?

Answer: In some areas, the sample Checkstyle rules file (recommended by the SE-EDU guide) may be slightly more permissive/restrictive than (but not contradictory to) the given Java coding standard. You are free to tweak the Checkstyle file if you wish. It is fine as long as your code is compliant with the given coding standard.

 

Level-10: GUI

Warning

Attention Mac 🍎 users! Please use either Amazon Corretto or Azul Zulu JDK distribution (not any other JDK 21 distributions), if not, you might run into problems while doing Level-10.

Add a GUI to the chatbot. Use the JavaFX technology to implement the GUI.

Refer to the JavaFX tutorial @SE-EDU/guides to learn how to get started with JavaFX.

Please also note that we are using JDK 21 and not JDK 17.

Also, for JavaFX tutorial part 1 – Getting started, your build.gradle should be:

xxxxxxxxxx
plugins {
    id 'java'
    id 'application'
    id 'org.openjfx.javafxplugin' version '0.1.0'
    id 'com.github.johnrengelman.shadow' version '7.1.2'
}


repositories {
    mavenCentral()
}


javafx {
    version = '21.0.1'
    modules = ['javafx.controls', 'javafx.fxml', 'javafx.graphics']
}


application {
    mainClass.set("Launcher")
}


shadowJar {
    archiveBaseName = "javafx-tutorial-duke"
    archiveClassifier = null
}


java {
    sourceCompatibility = JavaVersion.VERSION_21
    targetCompatibility = JavaVersion.VERSION_21
}


tasks.withType(JavaCompile) {
    options.encoding = 'UTF-8'
}

Go through at least up to part 4 of the tutorial. Part 5 covers cosmetic UI tweaks, and is optional to learn.

Note

There are two non-trivial steps to take here:

  1. learning JavaFX basics

  2. creating a GUI for your chatbot

You are cautioned against trying to do both in one go. Instead, complete the JavaFX tutorial as a separate project before adding a GUI to the chatbot.

Warning

Common mistake: Forgetting to add a separate Launcher class (as explained in the JavaFX tutorial Part 1) when adding the GUI to your project.

Note

You no longer need to keep the text-based UI after adding a GUI (although you are welcome to). Similarly, there is no need to keep the I/O redirection style automated testing added via A-TextUiTesting) anymore -- that technique is suited for text UIs only.

Note

Question: Is the bye command still needed, now that the GUI can be closed in other ways? Answer: Yes, we recommend keeping it. Reason: Being able to close the app by typing a command is consistent with the app's CLI-style

A-UnitTesting: Write Unit Tests

Write at least 2 meaningful unit tests for your project using either JUnit or Mockito (your choice). Your tests should:

  1. Cover different functionalities in your code

  2. Include both positive and negative test cases

  3. Each test should have a clear purpose explained in a brief comment
